# 报告生成系统需求分析

## 1. 文档说明

本文基于 `report-generation/报告生成需求说明书.pdf` 整理，目标是把 PDF 中的原始需求转化为后续开发可使用的需求分析文档。

由于原 PDF 的文本层存在中文编码映射异常，本文根据可提取章节结构和可恢复文本进行整理。涉及无法完全确认的字段或行业词，已集中列入「待确认问题」。

## 2. 项目背景

系统目标是建设一个面向专业报告编写场景的智能报告生成系统。系统应覆盖从报告大纲生成、章节内容流式生成、在线编辑、记录管理到 Word 文档导出的完整流程，帮助用户高效完成专业报告编写工作。

PDF 明确要求系统支持两种固定类型的专业报告：

- 迎峰度夏检查报告
- 煤库库存审计报告

PDF 还明确强调：最终生成报告的完成度和效果将作为主要考核指标。

## 3. 业务目标

### 3.1 核心目标

- 支持用户基于固定报告类型快速生成专业报告。
- 支持 AI 生成报告大纲和章节内容。
- 支持用户在生成过程中持续编辑和修正内容。
- 支持把已生成和已编辑的报告导出为 DOCX Word 文档。
- 支持历史报告记录管理，方便重新下载或基于保存数据重新导出。
- 支持管理员维护报告模板、专业素材和大语言模型配置。

### 3.2 成功标准

- 生成报告结构完整，符合固定报告类型的模板要求。
- 报告内容可读、专业、可编辑。
- DOCX 导出结果样式统一，能保留用户编辑结果。
- 报告记录可追溯、可管理、可重复下载。
- 管理端可以维护模板、素材和模型配置。

## 4. 用户角色与权限

| 角色 | 权限范围 |
|------|----------|
| 未登录用户 | 只能访问登录/注册入口，不能访问系统功能和 API |
| 普通用户 | 登录后可生成报告、编辑报告、导出报告、查看历史报告记录 |
| 管理员 | 普通用户能力，加上模板管理、素材管理、模型配置、统计查看等后台能力 |
| 超级管理员 | 管理员能力，加上更高权限的系统配置和用户/权限管理能力，具体范围待确认 |

权限原则：

- 所有 API 请求都应进行身份认证。
- 未登录用户不能访问系统业务功能。
- 管理功能仅允许管理员和超级管理员访问。
- 用户登录后应维持会话状态，刷新页面后不需要重复登录。

## 5. 总体业务流程

### 5.1 用户侧报告生成流程

1. 用户注册或登录系统。
2. 用户选择报告类型。
3. 用户输入报告主题或基础信息。
4. 系统根据报告类型和预置模板生成固定结构的大纲。
5. 用户在线编辑大纲，包括删除章节、调整顺序等。
6. 系统对大纲重新编号。
7. 用户发起报告内容生成。
8. 系统按章节流式生成报告内容。
9. 用户实时查看生成进度和每个章节内容。
10. 用户在线编辑生成后的章节内容。
11. 系统保存报告数据和生成记录。
12. 用户导出 DOCX Word 文档。
13. 用户下载 Word 文档。

### 5.2 管理员侧维护流程

1. 管理员登录后台。
2. 管理报告模板，包含上传、查看、编辑、删除。
3. 管理专业素材文档，包含上传和维护。
4. 通过模板编辑器可视化配置模板结构。
5. 配置大语言模型参数，如 API 地址、密钥、模型名称、超时时间。
6. 查看报告生成相关统计指标和趋势图。

## 6. 功能需求

### FR-001 用户注册与登录

系统应支持用户注册和登录。

需求明细：

- 用户通过用户名和密码注册。
- 用户通过用户名和密码登录。
- 登录成功后系统创建并维持会话。
- 页面刷新后用户仍保持登录状态。
- 未登录用户访问业务页面或 API 时应被拒绝或重定向到登录页。
- 所有业务 API 都应校验用户身份。

验收标准：

- 未登录状态访问报告生成功能会被拦截。
- 登录后刷新页面仍保持登录状态。
- 登出后再次访问业务 API 会失败。

### FR-002 报告类型管理与选择

系统应支持两种固定报告类型。

需求明细：

- 报告类型 1：迎峰度夏检查报告。
- 报告类型 2：煤库库存审计报告。
- 每种报告类型对应预定义模板和固定大纲结构。
- 用户生成报告前应选择报告类型。
- 系统根据报告类型决定大纲结构、模板、内容生成约束和导出样式。

验收标准：

- 用户可以在界面中选择两种报告类型。
- 不同类型生成的大纲结构不同，并与对应模板匹配。

### FR-003 报告大纲生成

用户输入报告主题后，系统应由 AI 驱动生成报告大纲。

需求明细：

- 用户输入报告主题或报告基础信息。
- 系统调用 AI，根据报告类型和模板生成大纲。
- 大纲支持多级章节结构，例如章、节、小节。
- 两种固定报告类型都应由预定义模板驱动生成固定大纲结构。
- 大纲生成后应展示在页面上。

验收标准：

- 输入主题后系统能生成结构化大纲。
- 大纲具有多级层级。
- 生成的大纲符合所选报告类型的模板。

### FR-004 大纲编辑

系统应允许用户在界面上编辑生成后的大纲。

需求明细：

- 用户可以删除章节。
- 用户可以调整章节顺序。
- 用户可以修改章节标题。
- 大纲编辑后系统应支持自动重新编号。
- 大纲编辑结果应作为后续内容生成的依据。

加分项：

- 支持对单个章节内容进行重新生成。

验收标准：

- 删除章节后编号能正确更新。
- 调整章节顺序后后续内容生成顺序同步变化。
- 修改章节标题后生成内容使用修改后的标题。

### FR-005 报告内容生成

系统应根据大纲逐章节流式生成报告内容。

需求明细：

- 系统按大纲章节逐步生成报告内容。
- 用户可实时查看每个章节的生成进度和内容。
- 系统展示总体生成进度，例如已完成章节数 / 总章节数。
- 每个章节生成完成后支持在线编辑和修改。
- 生成内容应支持正文内容编写。
- 生成内容应支持表格数据处理与呈现。

验收标准：

- 生成过程中用户可以看到当前进度。
- 每个章节生成后可以查看内容。
- 用户可以编辑已生成章节。
- 表格类内容在页面中能正常呈现。

### FR-006 报告在线编辑

系统应支持用户对生成后的报告内容进行编辑。

需求明细：

- 用户可以修改章节正文。
- 用户可以调整或补充表格内容。
- 编辑结果应被保存。
- 后续导出 DOCX 时必须使用用户编辑后的内容。

验收标准：

- 用户编辑章节后刷新或重新进入记录，编辑内容不丢失。
- 导出的 Word 文档包含编辑后的最终内容。

### FR-007 Word 文档导出

系统应支持将生成后的报告导出为 DOCX 格式。

需求明细：

- 系统将生成报告导出为 Word 文档，格式为 DOCX。
- 导出的文档应保留用户在流式生成过程中的编辑修改。
- 用户可以下载已生成的报告文件。
- 系统应支持基于已保存报告数据重新生成 Word 文档。
- 重新生成 Word 文档时不需要重新执行 AI 内容生成。

加分项：

- DOCX 文档遵循后台统一配置的样式体系。
- 样式包括但不限于字体、字号、标题层级、段落间距、表格样式、页眉页脚。
- 确保所有报告输出风格一致。

验收标准：

- 用户可成功下载 DOCX 文件。
- DOCX 文件能用 Word 打开。
- DOCX 内容与页面最终编辑内容一致。
- 对同一报告记录重新导出 DOCX 时，不重新调用 AI。

### FR-008 报告记录管理

系统应记录每次报告生成记录。

需求明细：

- 记录报告名称。
- 记录报告类型。
- 记录专业。
- 记录电厂或业务对象字段，具体含义待确认。
- 记录年份。
- 记录生成时间。
- 记录状态。
- 报告记录列表支持分页浏览。
- 用户可以查看历史报告记录。
- 用户可以基于保存数据重新生成 Word。
- 用户可以重新下载已生成文件。
- 用户可以删除不再需要的报告记录。

验收标准：

- 生成报告后列表中出现一条记录。
- 记录包含关键元数据。
- 历史记录可以重新下载。
- 删除记录后列表不再展示该记录。

### FR-009 报告模板管理

系统应支持上传和管理报告模板文件。

需求明细：

- 管理员可以上传报告模板文件。
- 管理员可以查看模板列表。
- 管理员可以删除模板。
- 模板应与报告类型关联。
- 模板应支持通过编辑器进行可视化编辑和配置。

验收标准：

- 管理员可以上传模板。
- 模板上传后可在列表中查看。
- 删除模板后不可再被选择或使用。

### FR-010 素材管理

系统应提供素材管理功能。

需求明细：

- 管理员可以上传专业素材文档。
- 管理员可以管理专业素材文档。
- 素材可作为报告生成时的参考资料，具体引用方式待确认。

验收标准：

- 管理员可以上传素材。
- 上传后的素材可被查看和管理。
- 素材是否参与 AI 生成，需要在后续设计中明确。

### FR-011 模板编辑器

系统应提供模板编辑器。

需求明细：

- 管理员可以对模板结构进行可视化配置。
- 模板编辑器应支持章节结构配置。
- 模板编辑器应支持与固定报告类型绑定。
- 是否支持样式配置、占位符配置、变量配置，需要后续确认。

验收标准：

- 管理员可以打开模板编辑器。
- 管理员可以修改模板结构并保存。
- 修改后的模板能影响后续大纲或导出结果。

### FR-012 大语言模型配置

系统应提供大语言模型配置页面。

需求明细：

- 管理员可在运行时修改模型 API 地址。
- 管理员可配置 API 密钥。
- 管理员可配置模型名称。
- 管理员可配置超时时间。
- 模型配置应仅管理员或超级管理员可访问。
- API 密钥等敏感信息应加密或至少脱敏展示。

验收标准：

- 管理员可以保存模型配置。
- 普通用户不能访问模型配置页面。
- 密钥字段不会在页面或日志中明文泄露。

### FR-013 数据统计与监控

系统应提供管理后台概览页面。

需求明细：

- 展示报告生成相关核心指标。
- 至少包含报告模板数量。
- 至少包含报告生成记录数量。
- 展示近 30 天报告生成趋势图表。

验收标准：

- 管理员可以查看统计概览页。
- 统计数据与实际记录数量一致。
- 趋势图能反映近 30 天生成数量变化。

## 8. 非功能需求

### 8.1 安全性

- 所有 API 必须鉴权。
- 管理功能必须做角色鉴权。
- API 密钥、模型密钥等敏感信息不能明文返回前端。
- 用户密码必须哈希存储。
- 文件上传应限制类型、大小和存储路径。

### 8.2 可用性

- 生成过程应展示进度，避免用户误以为系统卡死。
- AI 生成失败时应给出明确错误提示。
- 长时间生成任务应支持状态查询。
- 刷新页面后用户登录状态和生成记录应保持。

### 8.3 可维护性

- 报告类型、模板、大纲结构应解耦，方便后续新增报告类型。
- DOCX 样式配置应集中管理，避免散落在导出代码中。
- 模型配置应可运行时更新，避免每次改模型都重新发布系统。

### 8.4 性能

- 生成过程应采用流式输出或分章节异步生成。
- 报告列表应分页。
- 统计接口应避免全表实时重算，可考虑缓存或聚合。

### 8.5 可追溯性

- 每次报告生成应记录发起人、时间、类型、状态。
- 生成失败应记录失败原因。
- 模板和模型配置变更建议记录操作日志。

## 9. 推荐 MVP 范围

### 9.1 第一阶段必须实现

- 用户登录与会话保持。
- 两种固定报告类型选择。
- 基于模板生成报告大纲。
- 大纲展示和基础编辑。
- 按章节生成报告内容。
- 生成进度展示。
- 章节内容在线编辑。
- 报告记录保存。
- DOCX 导出和下载。
- 基础报告记录列表。

### 9.2 第一阶段可简化

- 模板编辑器先做结构配置，不做复杂拖拽。
- 素材管理先做上传和列表，不一定立即接入 AI 检索。
- 统计页先展示模板数量、报告数量、近 30 天趋势。
- DOCX 样式先实现一套默认样式，后续再后台可配置。

### 9.3 后续增强

- 单章节重新生成。
- 完整可视化模板编辑器。
- 专业素材检索增强生成。
- 多模型配置和模型切换。
- 生成质量评分和人工审核流程。
- 报告版本管理。

## 10. 关键风险

| 风险 | 影响 | 建议 |
|------|------|------|
| PDF 原始需求中部分字段语义不清 | 可能导致数据模型字段设计偏差 | 先列为待确认项，开发前与业务确认 |
| AI 生成质量不可控 | 影响主要考核指标 | 引入模板约束、素材约束、章节级编辑和人工确认 |
| DOCX 导出样式复杂 | 可能导致交付效果不稳定 | 先固定样式体系，再做后台可配置 |
| 模板编辑器范围过大 | 容易拖慢 MVP | 第一阶段只做必要结构配置 |
| 模型 API 密钥配置涉及安全 | 可能造成敏感信息泄露 | 加密存储、脱敏展示、限制权限 |
| 长报告生成耗时较长 | 用户体验差 | 分章节生成、展示进度、保存中间结果 |

### 导出与记录

- [ ] 生成后的报告可以导出 DOCX。
- [ ] DOCX 保留用户编辑内容。
- [ ] 用户可以下载 DOCX。
- [ ] 系统保存报告记录。
- [ ] 用户可以分页查看历史报告。
- [ ] 用户可以重新下载历史报告文件。
- [ ] 用户可以删除报告记录。

- [ ] 管理员可以查看、编辑、删除模板。

## 13. 建议开发拆分

### 任务 1：认证与权限基础

- 用户注册、登录、会话保持。
- API 鉴权中间件。
- 管理员权限控制。

### 任务 2：报告类型与模板基础

- 固定报告类型枚举。
- 模板数据结构。
- 基础模板上传和列表。

### 任务 3：大纲生成与编辑

- 主题输入。
- AI 大纲生成。
- 多级大纲展示。
- 删除、排序、重编号。

### 任务 4：内容生成与在线编辑

- 分章节生成。
- 生成进度展示。
- 章节内容编辑保存。

### 任务 5：报告记录与 DOCX 导出

- 报告记录列表。
- DOCX 生成。
- 下载和重新导出。

### 任务 6：管理后台增强

- 素材管理。
- 模型配置。
- 统计概览。

## 14. 结论

该需求的核心不是单纯生成文本，而是一个围绕固定报告类型的完整报告生产工作流。MVP 应优先打通「登录 - 选择报告类型 - 生成大纲 - 编辑大纲 - 生成内容 - 编辑内容 - 保存记录 - 导出 Word」闭环。模板编辑器、素材管理、模型配置和统计监控属于支撑能力，可以按团队开发节奏逐步完善。
